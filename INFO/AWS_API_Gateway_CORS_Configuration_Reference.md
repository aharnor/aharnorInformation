<h1>AWS API Gateway CORS Configuration Reference</h1><p>This document serves as the definitive reference for the working Cross-Origin Resource Sharing (CORS) configuration established for the Twin API. It details the architecture, the exact configuration parameters, the command-line interface (CLI) commands used to apply them, and why this design is superior to handling CORS within the backend application code (<code>server.py</code>).</p><hr><h2>1. Architectural Overview</h2><p>In modern web applications, browser security (Same-Origin Policy) requires that a frontend hosted on one domain (e.g., an S3 static website bucket) must negotiate permission before communicating with an API hosted on another domain (e.g., AWS API Gateway). This negotiation happens via a <strong>CORS Preflight</strong> handshake.</p><p>There are two primary ways to handle this handshake:</p><pre class="bg-muted rounded-md p-4 font-mono text-sm overflow-x-auto"><code>Approach A: Application-Level CORS (Ed's likely next step)
[Browser] ---&gt; [API Gateway (Pass-through)] ---&gt; [Lambda (server.py handles OPTIONS &amp; Headers)]

Approach B: Gateway-Level CORS (Our current working setup)
[Browser] ---&gt; [API Gateway (Intercepts OPTIONS &amp; injects headers)] --(POST only)--&gt; [Lambda (server.py)]</code></pre><h3>Why Gateway-Level CORS is Superior</h3><ol class="tight" data-tight="true"><li><p><strong>Zero Cold Starts for Preflights:</strong> Browser preflight (<code>OPTIONS</code>) requests are answered instantly by API Gateway in milliseconds. They never invoke your Lambda function, saving execution costs and avoiding Lambda cold-start latency.</p></li><li><p><strong>Cleaner Backend Code:</strong> Your <code>server.py</code> does not need to be cluttered with CORS middleware, manual header injections, or routing logic for the <code>OPTIONS</code> HTTP method.</p></li><li><p><strong>Centralized Security:</strong> CORS policies are managed at the entry point of your infrastructure rather than being scattered across individual microservices or Lambda functions.</p></li></ol><hr><h2>2. The Working API Gateway Configuration</h2><p>The API Gateway is configured as an <strong>HTTP API</strong> (Protocol Type: <code>HTTP</code>) in the <code>eu-north-1</code> region.</p><h3>CORS Configuration Parameters</h3><p>The gateway is configured with the following strict CORS rules:</p><p>ParameterValuePurpose<strong>AllowOrigins</strong><code>["*"]</code>Allows requests from any origin (including your S3 bucket).<strong>AllowMethods</strong><code>["GET", "POST", "OPTIONS"]</code>Restricts allowed HTTP methods to only those used by the application.<strong>AllowHeaders</strong><code>["content-type", "authorization", "x-requested-with"]</code>Explicitly lists allowed headers. <strong>Wildcards (</strong><code>*</code><strong>) are rejected by browsers here.AllowCredentials</strong><code>false</code>Disabled to allow the wildcard origin (<code>*</code>) to function securely.<strong>MaxAge</strong><code>300</code>Tells the browser to cache this preflight permission for 5 minutes (300 seconds) to reduce network requests.</p><hr><h2>3. Route Mappings</h2><p>To prevent API Gateway from forwarding preflight requests to Lambda, we explicitly separated the routes. API Gateway only triggers the Lambda integration for actual data-carrying requests.</p><p>Route KeyTarget IntegrationDescription<code>GET /integrations/gg87emp</code>Serves the root endpoint via Lambda.<code>POST /chatintegrations/rs3p6l9</code>Routes chat messages to the backend Lambda.<code>POST /{proxy+}integrations/6x82uiv</code>Catch-all route for other POST requests.</p><p><em>Note: Because there is no </em><code>ANY</code><em> or </em><code>OPTIONS</code><em> route defined, API Gateway's built-in CORS engine automatically intercepts all incoming </em><code>OPTIONS</code><em> requests, matches them against the CORS configuration, and returns a </em><code>204 No Content</code><em> response directly to the browser.</em></p><hr><h2>4. Exact CLI Commands Used</h2><p>These AWS CLI commands can be used to recreate or audit this configuration at any time.</p><h3>Step 1: Apply the CORS Configuration to the API</h3><p>This command configures the gateway's automatic CORS engine with the explicit headers and methods required by the browser:</p><pre class="bg-muted rounded-md p-4 font-mono text-sm overflow-x-auto"><code class="language-bash">aws apigatewayv2 update-api \
  --api-id 6pc8dpxa07 \
  --region eu-north-1 \
  --cors-configuration '{
    "AllowCredentials": false,
    "AllowHeaders": ["content-type", "authorization", "x-requested-with"],
    "AllowMethods": ["GET", "POST", "OPTIONS"],
    "AllowOrigins": ["*"],
    "MaxAge": 300
  }'</code></pre><h3>Step 2: Clean Up Wildcard Routes (If Recreating)</h3><p>If an <code>ANY /{proxy+}</code> or explicit <code>OPTIONS</code> route exists, it will bypass the gateway's CORS engine. They must be removed or restricted to <code>POST</code> or <code>GET</code> only:</p><pre class="bg-muted rounded-md p-4 font-mono text-sm overflow-x-auto"><code class="language-bash"># Delete an explicit OPTIONS route if it was manually created
aws apigatewayv2 delete-route \
  --api-id 6pc8dpxa07 \
  --route-id ih0zr4f \
  --region eu-north-1

# Update a wildcard route from ANY to POST
aws apigatewayv2 update-route \
  --api-id 6pc8dpxa07 \
  --route-id 52cwtwm \
  --route-key "POST /{proxy+}" \
  --region eu-north-1</code></pre><hr><h2>5. Verification and Troubleshooting</h2><p>To verify that the configuration is working independently of browser caching issues, use the following <code>curl</code> commands.</p><h3>Test 1: Preflight Handshake (<code>OPTIONS</code>)</h3><p>Simulate the browser's preflight check. It must return HTTP status <code>204</code> with the <code>access-control-allow-*</code> headers:</p><pre class="bg-muted rounded-md p-4 font-mono text-sm overflow-x-auto"><code class="language-bash">curl -v -X OPTIONS \
  -H "Origin: http://twin-frontend-930458520501.s3-website-eu-north-1.amazonaws.com" \
  -H "Access-Control-Request-Method: POST" \
  -H "Access-Control-Request-Headers: content-type" \
  https://6pc8dpxa07.execute-api.eu-north-1.amazonaws.com/chat</code></pre><h3>Test 2: Actual Request (<code>POST</code>)</h3><p>Simulate the actual payload delivery. It must return HTTP status <code>200</code> along with the backend's JSON response and the <code>access-control-allow-origin: *</code> header:</p><pre class="bg-muted rounded-md p-4 font-mono text-sm overflow-x-auto"><code class="language-bash">curl -v -X POST \
  -H "Origin: http://twin-frontend-930458520501.s3-website-eu-north-1.amazonaws.com" \
  -H "Content-Type: application/json" \
  -d '{"message": "hello"}' \
  https://6pc8dpxa07.execute-api.eu-north-1.amazonaws.com/chat</code></pre>
